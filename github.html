Github

Git is the industry-standard version control system for web developers. GitHub is both a website and a service that facilitates software development by allowing you to store your code in containers, called repositories (Git repository = project managed), and by tracking changes made to your code. In addition, it offers a hosting service and tools to build, test, and deploy code. —> GitHub is a tool for working with Git —> Git is Tool, Github is service


Structure

  • working directory: where all the work is done
  • staging area: where you’ll list changes you make to the working directory
  • repository: where git permanently stores those changes as different versions of the project
  • branch: divergence from the main project. When you branch out, git is essentially making a new state of your current code, upon which you can work, without affecting the important main state of the code. One can create as many branches as they wish and even create branches off of other branches.
    —> main branch is where changes get merged into.
    →create new branch when fixing bug, adding feature
    →naming convention with logic, bsp: `carlos_feature_dashboard_notifications` (name_branchtype_short branch description)
    clone=download repository → create new branch → commit
  • pull requests - allow collaborators to review and give feedback on proposed code changes before they are merged to the main branch

Commands

  • git init - (re)initialize git project
  • git add filename / git add . - um neue Datei in die staging area (Bearbeitungsraum) zu rufen (auch nach commit muss wieder geadded werden). add . für alle Files, ansonsten git add filename1 filename2
  • git status - überprüft, ob changes gemacht wurden (modified), untracked means that Git sees the file but has not started tracking changes yet.
  • git diff filename - vergleicht änderung(nicht gespeichert) mit original und zeigt Unterschiede (grün)
  • git commit -m "Bezeichnung Commit" - permanently stores changes from the staging area inside the repository- Head Commit is commit you're on
  • git commit --amend - to update=replace your previous commit, to keep same -m write $ git commit --amend --no-edit
  • git show head - zeigt head commit an
  • git checkout HEAD filename - git checkout HEAD filename
  • git log - lists commits made; git log --oneline shows commit in one line format. git log -S "keyword" displays a list of commits that contain the keyword in the message. git log --oneline --graph displays a visual representation of how the branches and commits were created in order to help you make sense of your repository history. When used alone, the description can be very lengthy, so you can combine the command with --oneline in order to shorten the description.
  • git reset HEAD filename - resets file to be the same as in HEAD commit → It does not discard file changes from the working directory, it just removes them from the staging area
  • git reset commit_SHA - sets head commit to the commit with the SHA beginning with the 6 digits, bsp git reset 5d69206
  • git stash - aktuelles Projekt pausieren, um was anderes zu bearbeiten: projekt wird in ablage verfrachtet und per git stash pop wieder aktiviert
  • git alias commands - commands kürzel benennen mit $ git config --global alias.co "checkout" wird zu $ git co example_branch (co anstatt checkout)

An Branch weiterbasteln

  1. Branch(=code) kopieren und zb in GitBash weiterarbeiten
  2. Dafür GitBash/Terminal öffnen, per cd & ls passenden Ordner auswählen und git clone LINK eingeben
  3. Dann neuen Branch erstellen durch git checkout -b NEWBRANCHNAME
  4. git add newbranchname
  5. git commit -m “Added new print”
  6. um neuen branch hochzuladen: git push - - set-upstream origin NEWBRANCHNAME
  7. —> create pull request, um andere teammember dazu anzuregen den eigenen neuen branch abzusegnen, um ihn zu aktivieren; good practice als Kommentar zu schreiben “to test: Make sure that xxx. Thanks!”


README.md

text file that is meant to be read as soon as someone views the repository. This file contains text to introduce, explain, and share the information required to understand what the project is about. - often as .md file

Include:

  • Project Title
  • Description: describe the main purpose of your project. why did you build this project? and what problem(s) does it solve?
  • Features: If your project has multiple features, list them here. You can even add screenshots and gifs to show off the features.
  • How to use: write step-by-step instructions on how to install and use your project. Any software or package requirements should also be listed here.
  • Technologies: List all the technologies and/or frameworks you used and what purpose they serve in your project.
  • Collaborators: Write your team members’ or collaborators’ names here along with a link to their GitHub profile.
  • License: list a license on your README so other developers can understand what they can and cannot do with your project. You can use this guide to help you choose a license.
